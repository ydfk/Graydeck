# Mihomo Docker 镜像设计方案

## 1. 目标

设计一个面向个人或小型家庭网络环境的 `mihomo` Docker 镜像，提供以下能力：

- 一个简单、稳定、易维护的管理页面
- 主要覆盖 `mihomo` 配置管理与核心自动更新
- 内置 `zashboard` UI，并作为专业操作面板提供完整能力
- 支持对 `zashboard` 进行功能裁剪或屏蔽
- 保持镜像部署简单，优先单镜像方案
- 管理页视觉严格遵循 [web/DESIGN.md](F:\github-my\mihomo\web\DESIGN.md) 的色彩、排版、阴影边框与圆角规范

## 2. 非目标

当前阶段先不做以下内容：

- 不开发桌面客户端或移动端原生应用
- 不做多用户、RBAC、组织协作
- 不做订阅市场、节点分享社区、远程云托管
- 不直接改造 `zashboard` 内部视觉风格
- 不把所有 `mihomo` API 能力重新在管理页里完整复制一遍

## 3. 设计原则

### 3.1 产品原则

- 管理页做“少而准”的高频操作
- 复杂实时流量、连接、策略组操作交给 `zashboard`
- 默认路径清晰，尽量减少用户理解成本
- 所有危险动作都要有校验、确认、回滚
- 管理页优先解决部署后最常用的配置、更新、回滚问题

### 3.2 技术原则

- 优先单镜像、单入口、少进程
- 将 `mihomo` 外部控制接口限制在容器内部
- 管理页与 `zashboard` 统一由同一入口服务暴露
- 更新能力与运行能力解耦，避免一次失败更新拖垮运行态
- `zashboard` 作为可裁剪面板集成，而不是完全裸露原始功能

### 3.3 视觉原则

- 界面 chrome 严格黑白
- 彩色只允许出现在“产品内容区”，例如流量图、版本趋势、嵌入式内容预览
- 交互组件统一使用 `50px` pill 或 `50%` circle
- 小容器只使用 `6px`，卡片/弹层只使用 `8px`
- 焦点态统一 `dashed 2px`

## 4. 推荐总体方案

推荐采用“单镜像 + 单管理守护进程 + 内嵌静态前端 + 子进程启动 mihomo”的结构。

### 4.1 镜像内组件

- `mihomo` 核心二进制
- `managerd`
  - 负责启动/停止/重启 `mihomo`
  - 提供管理 API
  - 提供配置校验、备份、回滚、更新编排
  - 反向代理 `mihomo external-controller`
  - 托管管理页静态资源
  - 托管 `zashboard` 静态资源
- 管理页前端静态文件
- `zashboard` 静态文件

### 4.2 为什么不推荐“三服务拼装式单镜像”

不推荐在一个镜像里再塞 `nginx/caddy + node/go API + supervisor + mihomo` 四层结构，原因如下：

- 进程过多，故障定位复杂
- 更新链路变长
- 配置管理页面本身不需要独立高性能反向代理
- 后续要做核心热更新与状态治理时，统一由 `managerd` 接管更清晰

## 5. 推荐运行形态

### 5.1 Linux 主推荐

对需要 `TUN` 或透明代理能力的用户，推荐：

- `network_mode: host`
- `cap_add: NET_ADMIN`
- 挂载 `/dev/net/tun`

这样能最大程度兼容 `mihomo` 常见运行场景。

### 5.2 轻量模式

对仅需要 HTTP/SOCKS 代理和 Web 管理的用户，可支持桥接网络模式，但不作为主设计路径。

## 6. 信息架构

推荐管理页只做 5 个一级页面。

### 6.1 `/` 总览

用于替代“用户一上来就直接面对一整套 `zashboard`”的复杂感。

展示内容：

- 核心运行状态
- 当前配置来源与最后应用时间
- 代理端口与控制端口状态
- 当前版本 / 可更新版本
- 最近一次更新结果
- 最近一次配置校验结果
- 快捷动作
  - 打开配置管理
  - 检查核心更新
  - 重启核心
  - 打开 `zashboard`

### 6.2 `/config` 配置管理

这是管理页的核心页面。

功能范围：

- 当前启用订阅源与最后同步状态
- 订阅源列表管理
- 当前配置来源
  - 远程订阅 URL
  - 本地文件
  - 上传导入
- 当前生效配置摘要
- 在线配置编辑
- 草稿保存
- 上传配置文件
- 从远程订阅拉取配置
- 配置同步策略设置
- 导出当前配置
- 配置语法校验
- 运行前检查
- 与当前版本 diff 对比
- 应用配置
- 回滚到历史版本
- 查看变更历史
- 敏感字段遮罩显示

推荐拆成 4 个区域：

- 左侧：订阅源列表、当前启用订阅、同步状态、最近版本、备份列表
- 中间：编辑器 / 上传面板
- 右侧：校验结果与变更摘要
- 底部：应用、回滚、下载备份等操作区

推荐补充两个二级视图：

- `草稿`
- `历史`

### 6.3 `/updates` 更新中心

只聚焦两个对象：

- `mihomo` 核心更新
- `zashboard` 内置资源版本

页面内容：

- 当前核心版本
- 更新通道
  - `Meta`
  - `Alpha`
- 目标架构识别结果
- 新版本检测结果
- 自动更新时间策略
- 最近更新日志
- 一键更新与回滚入口

### 6.4 `/system` 系统与日志

展示：

- `mihomo` 运行日志
- `managerd` 操作日志
- 容器路径与挂载状态
- 核心 PID、启动时间、重启次数
- 健康检查状态

### 6.5 `/zashboard`

提供内置 `zashboard`。

此页面只做轻包装，不重新实现内部功能。推荐：

- 顶部保留一个极薄的壳层标题栏
- 提供“返回总览”“全屏打开”“重新连接 API”三个动作
- 内容区域直接承载 `zashboard`
- 根据配置模式动态隐藏或禁用部分能力

## 7. 管理页与 zashboard 的关系

### 7.1 职责分工

管理页负责：

- 配置编辑与发布
- 核心更新
- 历史版本与回滚
- 容器内运行态治理
- 简化后的状态总览

`zashboard` 负责：

- 实时连接视图
- 节点与策略组管理
- 延迟测试
- 流量明细
- 规则视图
- 实时日志联动

### 7.2 集成方式

推荐将 `zashboard` 作为镜像内静态资源打包，暴露在：

- `/zashboard/`

不建议首次版本就深度魔改 `zashboard`，原因如下：

- 升级跟随成本高
- 会放大前端维护量
- 容易与官方/上游发布节奏脱节

### 7.3 与 mihomo API 的连接方式

推荐不直接把 `mihomo external-controller` 暴露给公网，而是：

- `mihomo` 仅监听容器内部回环地址
- `managerd` 提供受控代理路径，例如 `/api/clash`
- `zashboard` 通过同域地址访问代理路径

这样可以减少：

- `secret` 暴露风险
- 端口暴露数量
- 跨域和同源策略问题

### 7.4 zashboard 功能裁剪策略

你提到“有可能想屏蔽一些功能”，建议首版就按“可裁剪集成”设计。

推荐分 3 档模式：

- `full`
  - 完整开放 `zashboard`
- `safe`
  - 保留状态查看、日志、连接、延迟测试、策略组切换等常用能力
  - 屏蔽高风险写操作或重复配置入口
- `minimal`
  - 只保留基础状态查看、跳转和少量只读能力

### 7.5 推荐屏蔽范围

如果首版要稳妥，推荐默认采用 `safe` 模式。

优先考虑屏蔽的功能类型：

- 直接修改运行配置的入口
- 可能改变代理行为的批量操作
- 暴露敏感配置或密钥的展示入口
- 与管理页已有能力重复的配置管理入口

优先保留的功能类型：

- 连接与流量观测
- 节点延迟测试
- 策略组切换
- 实时日志查看
- 规则命中观察

### 7.5.1 `safe` 模式保留功能清单

建议在 `safe` 模式下明确保留以下能力：

- 仪表盘首页与运行状态查看
- 代理节点与策略组列表查看
- 策略组切换
- 节点或节点组延迟测试
- 实时流量图表与速率查看
- 连接列表查看、筛选、复制
- 日志查看
- 规则命中与规则列表查看
- 通过壳页面执行“重新连接 API”

其中只有以下写操作允许保留：

- 策略组切换

原因：

- 这是 `zashboard` 最核心的日常使用价值之一
- 风险明显低于直接改配置、升级核心或切换运行模式

### 7.5.2 `safe` 模式屏蔽功能清单

建议在 `safe` 模式下明确屏蔽以下能力：

- `setup` 页面中的连接参数手动录入与修改
- 核心升级按钮与自动升级入口
- `TUN` 开关
- 与管理页重复的配置管理入口
- 可能暴露 `secret`、控制地址或原始连接信息的设置入口
- 可能直接修改运行配置的入口
- 可能触发重启、重载、模式切换的高风险入口
- 与 UI 资源升级相关的入口

### 7.5.3 `safe` 模式技术控制点

`safe` 模式建议通过 3 层同时生效：

- 第一层：外层壳页面隐藏导航与入口
- 第二层：路由级限制，阻止进入不允许的页面
- 第三层：`/api/clash/*` 代理层按方法和路径做 allowlist / denylist

推荐最小控制规则：

- 允许只读查询接口通过
- 允许策略组切换所需接口通过
- 阻断配置修改、核心升级、运行模式切换相关接口

另外建议利用 `zashboard` 官方支持的 URL 参数：

- `disableUpgradeCore=1`
- `disableTunMode=1`

这两个参数可以直接隐藏升级按钮和 `TUN` 开关，适合作为首层控制。

### 7.6 实现思路

首版不建议深改 `zashboard` 源码逻辑，优先采用低侵入策略：

- 通过路由入口控制，只开放允许访问的页面
- 通过反向代理规则阻断不允许的 API 写操作
- 在外层壳页面隐藏不希望用户进入的导航入口

如果后续发现仅靠外层控制不够，再评估维护一个轻量 fork。

### 7.7 `safe` 模式最终结论

首版 `safe` 模式的最终目标是：

- 保留“看状态、看连接、看日志、测速、切策略组”
- 屏蔽“改配置、升核心、切 `TUN`、改控制连接、暴露敏感信息”

这样既保留了 `zashboard` 的核心实用价值，也不会和你自己的管理页职责冲突。

## 8. 核心配置与数据目录设计

推荐统一使用 `/data` 作为持久化根目录。

### 8.1 目录规划

- `/data/config/config.yaml`
- `/data/config/history/`
- `/data/core/`
- `/data/ui/zashboard/`
- `/data/logs/`
- `/data/tmp/`

说明：

- `config.yaml` 为当前生效配置
- `history/` 存放带时间戳的历史版本
- `core/` 存放已下载核心版本与当前软链接
- `ui/zashboard/` 预留未来可选的 UI 资源更新能力

## 9. 配置管理流程设计

### 9.0 配置管理能力边界

配置管理模块建议覆盖以下能力：

- 以远程订阅为主的配置更新方式
- 支持多个订阅源，但同一时间只启用一个
- 区分“草稿”和“生效配置”
- 支持多种配置来源
- 支持版本历史与回滚
- 支持校验、预检查与失败保护
- 支持敏感字段遮罩
- 支持配置摘要与结构化预览

首版不建议做的高复杂能力：

- 可视化拖拽式全量配置生成器
- 在线规则市场与远程模板仓库
- 多配置并行运行

### 9.1 配置对象模型

建议将配置对象统一抽象为：

- `sourceType`
- `draftContent`
- `activeContent`
- `versionId`
- `createdAt`
- `updatedAt`
- `validationStatus`
- `applyStatus`

其中：

- `sourceType` 取值为 `local`、`upload`、`remote`
- `draftContent` 表示尚未生效的编辑内容
- `activeContent` 表示当前运行中的正式配置

### 9.2 配置来源设计

建议首版支持 3 类来源：

- 远程订阅 URL 拉取
- 本地已有配置
- 本地上传导入

其中主推荐路径为：

- 远程订阅 URL

本地文件和上传导入作为补充手段，而不是默认主路径。

远程订阅模式下建议保存：

- 订阅 ID
- 源地址
- 订阅名称
- 是否为当前启用订阅
- 是否启用自动同步
- 自动同步频率
- 自动应用策略
- 最近拉取时间
- 最近成功版本
- 最近失败原因
- 是否允许手动覆盖本地草稿

并明确约束：

- 可以保存多个订阅源
- 同一时间只能有一个“当前启用订阅”
- 当前生效配置默认与当前启用订阅关联
- 切换启用订阅不等于立即应用其内容

### 9.2.1 远程订阅主流程

推荐将远程订阅更新设计为配置管理的默认主路径：

1. 用户填写订阅 URL
2. 系统立即拉取一次远程配置
3. 执行语法校验与运行前检查
4. 生成与当前生效配置的 diff
5. 用户确认后应用，或仅保存为草稿
6. 后续按同步策略定时检查更新

如果系统内存在多个订阅源，则先选择其中一个作为“当前启用订阅”。

### 9.2.2 订阅同步策略

建议首版支持以下策略：

- 手动检查更新
- 定时检查，发现变更后提示人工确认
- 定时检查，校验通过后自动应用

默认推荐：

- 定时检查，校验通过后自动应用

这样符合“当前启用订阅持续跟进”的主路径。

同步频率应允许用户在 UI 中修改。

### 9.2.3 订阅更新结果展示

每次订阅同步后建议明确展示：

- 是否发现变更
- 变更摘要
- 校验结果
- 是否已自动应用
- 失败原因
- 可回滚版本

### 9.2.4 多订阅源行为规则

多个订阅源场景下，建议明确如下规则：

- 当前启用订阅拉取到新版本后，如果目标就是当前正在应用的订阅链路，则直接执行校验
- 校验通过后直接应用
- 非当前启用订阅即使拉取到新内容，也只保留为候选版本，不自动切换
- 用户需要显式选择该订阅，并确认是否将其内容应用为新的生效配置

换句话说：

- 当前启用订阅可以自动推进
- 非当前启用订阅只能预拉取、预校验、预留存，不能抢占当前生效配置

### 9.3 草稿与生效配置

建议明确分成两层：

- 草稿层
- 生效层

交互规则：

- 用户编辑时只改草稿，不直接改生效配置
- 应用成功后，草稿升级为新生效配置
- 应用失败时，草稿保留，生效配置不变
- 订阅同步拉取的新内容默认先进入草稿，再按策略决定是否应用

这样可以避免误操作直接影响运行态。

### 9.4 编辑与应用流程

1. 用户编辑配置或上传配置
2. `managerd` 先落盘到临时文件
3. 调用 `mihomo` 对应校验流程
4. 校验通过后生成备份快照
5. 原子替换当前配置
6. 向 `mihomo` 发送重载或执行平滑重启
7. 记录操作日志

### 9.4.1 远程订阅更新流程

1. 定时任务或用户手动触发订阅检查
2. 拉取远程配置到临时文件
3. 与当前生效配置计算 diff
4. 执行语法校验与运行前检查
5. 判断该订阅是否为当前启用订阅
6. 如果是当前启用订阅，则校验通过后直接应用
7. 如果不是当前启用订阅，则只保留为候选版本并提示用户手动选择
8. 应用成功后写入历史版本
9. 失败时保留草稿并记录失败原因

### 9.5 校验与预检查

配置应用前建议至少经过两层检查：

- 语法校验
- 运行前检查

语法校验：

- `mihomo` 走 YAML 语法与核心校验命令

运行前检查：

- 必填入口端口是否合法
- 关键字段是否为空
- 引用的 provider、rule-set、geo 文件路径是否存在
- 远程资源地址是否格式正确
- 是否存在明显端口冲突

校验结果建议分级展示：

- 错误
- 警告
- 提示

### 9.6 回滚流程

1. 用户选择历史版本
2. 展示与当前配置的 diff
3. 二次确认
4. 恢复目标版本
5. 执行校验
6. 重载 `mihomo`

### 9.7 历史版本与审计

建议每次以下操作都生成历史记录：

- 导入配置
- 手动编辑后应用
- 远程拉取后应用
- 订阅自动同步后应用
- 非当前订阅拉取后留存候选版本
- 回滚

每条历史记录建议包含：

- 版本号
- 来源类型
- 创建时间
- 操作类型
- 操作结果
- 摘要说明

### 9.8 敏感字段处理

配置管理页需要对敏感内容做最小保护。

建议处理规则：

- 密钥类字段默认遮罩显示
- 提供“临时显示明文”动作
- 导出配置时允许用户选择是否保留敏感字段
- 日志中不打印完整密钥值

可纳入遮罩范围的典型字段：

## 10. 核心自动更新设计

### 10.1 设计边界

这里的“核心自动更新”指：

- 容器内 `mihomo` 二进制自动检查新版本并替换

这里不等价于：

- Docker 镜像自动更新

两者必须分开设计，否则职责会冲突。

### 10.2 推荐更新策略

推荐支持两种模式：

- 手动检查 + 手动更新
- 自动检查 + 用户确认后更新

首个版本不建议默认“无人值守自动重启式更新”，原因如下：

- 代理核心更新属于高风险操作
- 用户配置可能依赖某些分支行为
- Docker 用户通常也关心镜像不可变性

### 10.3 更新流程

1. 检测当前架构与通道
2. 拉取上游 release 元数据
3. 选择匹配的二进制资产
4. 下载到临时路径
5. 校验文件完整性
6. 解压并写入新版本目录
7. 切换当前版本软链接
8. 平滑重启 `mihomo`
9. 健康检查
10. 成功则记录版本，失败则自动回滚

### 10.4 回滚策略

至少保留最近 2 到 3 个核心版本，回滚时只切换软链接并重启。

### 10.5 通道设计

建议 UI 里明确区分：

- `Meta` 稳定通道
- `Alpha` 测试通道

并在页面上清楚提示：

- `Alpha` 更新更快
- `Meta` 更适合作为默认推荐

## 11. zashboard 更新策略

首期版本建议将 `zashboard` 当作“镜像内固定版本资源”，而不是默认开放在线自动升级。

原因：

- 你当前产品的主入口是自定义管理页
- 如果 `zashboard` 版本自由漂移，问题定位会更复杂
- 镜像测试矩阵会失控

推荐策略：

- 首发阶段内置固定版本
- 后台仅显示当前内置版本
- 后续版本再考虑“检查新 UI 资源包并替换”的可选能力

如果后续开放 `zashboard` 更新，建议也走与核心更新一致的：

- 下载
- 校验
- 版本留存
- 失败回滚

## 15. 技术方案建议

### 15.1 总体技术选型

推荐首版采用以下组合：

- 后端管理守护进程：Go
- 管理页前端：React + TypeScript + Vite 8+
- 前端本地状态：Zustand
- 前端服务端状态：TanStack Query
- 前端路由：React Router
- 包管理：pnpm
- 数据存储：SQLite + 文件系统
- 定时同步：Go 内部调度器

### 15.2 为什么推荐 Go

`managerd` 推荐使用 Go，原因如下：

- 与 `mihomo` 生态接近，二进制部署简单
- 进程管理、文件操作、反向代理都很适合
- 单镜像里不需要再引入额外运行时
- 配置校验、下载、解压、调度、日志处理可以统一在一个进程内完成

### 15.3 为什么推荐 React

管理页前端推荐使用 React，原因如下：

- 你已经明确希望管理页使用 React 生态
- 管理页是独立产品层，不需要和 `zashboard` 强行统一框架
- React 生态在表单、数据请求、编辑器集成、状态拆分方面很成熟
- 适合把“订阅管理、配置编辑、版本历史、更新中心”拆成清晰模块

这里的关键思路是：

- 管理页使用 React
- `zashboard` 保持其原生技术栈，作为内置独立 UI 通过路由嵌入

两者通过 `managerd` 的同域 API 串联，而不是在前端工程内强行混编。

### 15.4 为什么不优先选 Next.js

首版不优先选 Next.js，原因如下：

- 当前产品本质是容器内管理控制台，不需要 SSR
- 首版更适合静态资源构建后由 `managerd` 直接托管
- Vite 构建更轻，开发体验更直接
- 能减少首版工程复杂度

另外当前前端构建链路建议尽量走 Vite 8 的 Rust 方向能力：

- Rolldown 负责打包
- Oxc 负责更轻量的转换路径
- Lightning CSS 负责 CSS 处理与压缩

### 15.5 前端实现建议

管理页前端建议如下：

- React
- TypeScript
- Vite 8+
- React Router
- TanStack Query
- Zustand
- pnpm
- 原生 CSS 变量 + 少量 PostCSS

推荐职责划分：

- TanStack Query 管理订阅列表、运行状态、日志、更新状态这类服务端数据
- Zustand 管理局部 UI 状态，例如当前侧栏、过滤条件、弹层状态、编辑器模式

表单层建议：

- React Hook Form

编辑器建议：

- Monaco Editor

不建议首版引入重型 UI 组件库，原因如下：

- 你的视觉规范已经被 [web/DESIGN.md](F:\github-my\mihomo\web\DESIGN.md) 明确锁定
- 组件库往往会把界面带回通用后台风格
- 自定义黑白界面、字重层级、圆角尺度时，原生样式控制更直接

### 15.6 后端实现建议

`managerd` 建议拆成以下模块：

- `server`
  - HTTP API、静态资源托管、反向代理
- `subscription`
  - 订阅源管理、同步策略、候选版本处理
- `config`
  - 草稿、生效配置、历史版本、回滚
- `runtime`
  - `mihomo` 进程启动、停止、重启、健康检查
- `updater`
  - 核心版本检查、下载、回滚
- `zashboard`
  - `safe` 模式策略、路由控制、代理 ACL
- `storage`
  - SQLite 元数据与文件索引

### 15.7 数据存储建议

推荐采用“元数据进 SQLite，正文文件落磁盘”的组合：

- SQLite 保存订阅源、同步状态、历史索引、候选版本索引、操作日志
- YAML 文件保存生效配置、草稿配置、历史快照

这样做的好处：

- 结构化查询方便
- 配置正文仍然是直接可读、可导出的文件
- Docker 场景下备份与排障都比较友好

### 15.8 `zashboard safe` 的集成实现建议

推荐采用“静态资源嵌入 + 外层壳 + 代理 ACL”三段式集成：

1. 将 `zashboard` dist 资源打包进镜像，挂在 `/zashboard/`
2. 外层壳页面统一注入连接参数与 `safe` 模式限制
3. `managerd` 的 `/api/clash/*` 代理层对接口做访问控制

`safe` 模式至少要做：

- 自动附加 `disableUpgradeCore=1`
- 自动附加 `disableTunMode=1`
- 拦截配置修改相关请求
- 仅保留策略组切换这一类低风险写操作

### 15.9 `mihomo` 配置校验与应用建议

推荐使用“落临时文件 -> 调核心校验 -> 备份 -> 原子替换 -> 重载”的固定流程。

实现时建议：

- 校验优先调用 `mihomo` 自身的配置检查能力
- 校验通过后再写入正式配置
- 应用失败时立即回滚到上一成功版本
- 所有步骤都写操作日志

### 15.10 订阅同步技术建议

订阅同步建议由 `managerd` 内部调度器负责，不依赖额外 sidecar。

同步执行链路：

1. 读取当前启用订阅与同步频率
2. 定时拉取远程内容
3. 生成配置 diff
4. 调用校验与预检查
5. 如果是当前启用订阅则自动应用
6. 如果不是当前启用订阅则留存为候选版本
7. 更新数据库状态与历史记录

### 15.11 推荐首版开发边界

如果准备开始做，我建议首版只做这些：

- 单镜像 `managerd + mihomo`
- 管理页 5 个一级页面
- 多订阅源，但单启用订阅
- 当前启用订阅自动同步并自动应用
- `zashboard safe` 模式
- 配置历史与一键回滚

先不要在首版做：

- 自定义 fork `zashboard`
- 图形化配置生成器
- UI 在线升级
- 太细的权限系统
